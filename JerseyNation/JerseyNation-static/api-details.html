API Details

Technical integration scope for the external services referenced in the ERD and Data Dictionary.

About Open ERD

Integration Scope

The APIs listed below represent expected integrations necessary to support core business operations such as payments, shipping, taxation, and fraud detection. While these services are not yet implemented, the platform architecture has been designed to accommodate these integrations seamlessly in future versions.

Transactional and Operational APIs

Payment Gateway API
Processes payment authorization, capture, status callbacks, and refunds.
Connection type: Server-to-server REST integration over HTTPS
Typical methods: POST for authorize/capture/refund and GET for transaction lookup
Data format: JSON request and response payloads
Authentication: API key, secret token, or signed webhook validation
Used by: Payment block
Technical note: Usually supports synchronous responses for checkout and asynchronous webhooks for status updates.
Fraud / Risk API
Evaluates risk signals before or during payment authorization.
Connection type: Real-time REST scoring service
Typical methods: POST risk-evaluation request at checkout
Data format: JSON with order amount, device, address, customer, and behavioral signals
Authentication: API token or private service credential
Used by: Order block
Technical note: Common output includes risk score, decision flags, reason codes, and manual-review recommendation.
Tax Calculation API
Calculates taxes based on jurisdiction, product type, and destination.
Connection type: Server-side REST service
Typical methods: POST tax quote before order confirmation
Data format: JSON with line items, origin, destination, and taxable amounts
Authentication: API key or account token
Used by: Order block
Technical note: Often returns item-level and order-level tax breakdowns for auditability and invoicing.
Shipping Rate API
Quotes freight methods, service levels, and estimated delivery windows.
Connection type: REST integration with carrier or shipping aggregator
Typical methods: POST shipment quote request and GET service lookup
Data format: JSON with destination, parcel dimensions, weight, and service preferences
Authentication: API key or carrier account credential
Used by: Shipping block
Technical note: Usually returns service code, rate, estimated days, and carrier metadata for checkout display.
Carrier Tracking API
Retrieves shipment milestones and delivery status events after dispatch.
Connection type: Polling API and, in some cases, webhook/event subscription
Typical methods: GET tracking status by tracking number or shipment reference
Data format: JSON or XML depending on carrier
Authentication: Carrier token, account key, or signed callback
Used by: Shipping block
Technical note: Tracking events are often normalized internally into shipment milestones for consistent UI and reporting.
Address Validation API
Validates formatting, completeness, and deliverability of shipping addresses.
Connection type: Real-time REST validation service
Typical methods: POST address-validation request before shipment creation
Data format: JSON address object with normalized response fields
Authentication: API key
Used by: Shipping block
Technical note: Common outputs include normalized address lines, confidence score, and deliverability recommendations.

Commercial and Support APIs

Supplier / EDI API
Exchanges purchase orders, acknowledgements, catalog changes, and fulfillment signals with suppliers.
Connection type: REST, file-based integration, or EDI channel depending on supplier maturity
Typical methods: POST purchase orders, GET acknowledgements, or scheduled batch exchange
Data format: JSON, CSV, XML, or EDI documents such as ANSI X12 / EDIFACT
Authentication: Token, SFTP credential, VPN, or partner certificate
Used by: Supplier block
Technical note: Integration design often requires idempotency, document tracking, and reconciliation for procurement events.
Inventory Sync API
Synchronizes available quantity, reserved stock, and supplier-side availability.
Connection type: Scheduled REST synchronization or event-driven inventory updates
Typical methods: GET stock feed, POST stock adjustment, or webhook ingestion
Data format: JSON inventory records keyed by SKU or product identifier
Authentication: Token-based service authentication
Used by: Supplier block
Technical note: Usually requires timestamped updates and conflict rules to avoid stale stock levels across systems.
Email / CRM API
Supports customer communication, lifecycle triggers, and campaign delivery.
Connection type: REST messaging or CRM automation platform integration
Typical methods: POST campaign event, POST transactional email, GET delivery status
Data format: JSON with customer profile, segmentation attributes, and campaign metadata
Authentication: API token or OAuth 2.0
Used by: Marketing block
Technical note: Often integrates with templates, event triggers, suppression lists, and campaign performance feedback.
Recommendation API
Supports personalization and product suggestions based on customer behavior and product signals.
Connection type: REST inference service or internal model-serving endpoint
Typical methods: POST context request and GET ranked recommendation list
Data format: JSON with customer context, product context, and returned recommendation scores
Authentication: Service token or internal gateway credential
Used by: Marketing block
Technical note: May operate in real time for page rendering or in batch mode for campaigns and audience segments.
Analytics API
Collects product-view, campaign, and conversion events for reporting and performance analysis.
Connection type: Event collection endpoint, measurement API, or streaming ingestion service
Typical methods: POST event payloads and batch export ingestion
Data format: JSON event schema with timestamp, session, customer, and business attributes
Authentication: Measurement token, service key, or signed ingestion request
Used by: Marketing block
Technical note: A consistent event schema is critical to keep dashboards, attribution, and conversion metrics reliable.
No Mandatory External API
Some business domains remain internally managed in the current logical model.
Applies to: Customer, Product Catalog, and Administration areas
Connection type: Internal application services and direct database-backed workflows
Technical note: These domains can remain modular and still support future integration through adapters, service abstractions, or internal APIs if platform complexity grows.
These API descriptions extend the ERD by documenting the expected integration style, transport pattern, payload characteristics, and operational role of each external service in the JerseyNation logical architecture.
